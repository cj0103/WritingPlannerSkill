# Smart Doc Memory — 智能文档记忆与写作幕僚

## 核心定位

无 RAG（无向量数据库）的轻量级知识管理与写作助手。通过纯文本索引、长上下文理解和动态语义关联，实现"越用越聪明"的知识沉淀与材料起草。所有数据本地存储，无需外部服务。

**本技能是材料层**，产出知识卡片、索引、全局观和规则库。如需生成写作提示词（喂给大模型的 prompt），请使用 `writing-planner` 技能。两者分工：

```
smart-doc-memory（材料层）        writing-planner（提示词层）
─────────────────────────        ─────────────────────────
知识卡片、索引、全局观、规则  ──→  生成结构化写作提示词
                                    ↓
                              任意大模型 → 最终公文
```

## 当前知识库

本实例已初始化用户自建专题知识库（路径 `knowledge_base/`），包含：
- 结构化知识卡片（`cards/`）
- 全局索引（`index.json`）
- 全局观图谱（`macro_vision.md`）
- 写作规则库（`writing_rules.md`）

写作场景聚焦部署讲话、总结报告、实施方案、即兴简报四类公文。

## 知识库目录结构

```
knowledge_base/
├── cards/               ← 每份文档的独立知识卡片（一文档一MD）
├── index.json           ← 全局索引（含哈希，支持增量扫描）
├── macro_vision.md      ← 宏观全局观（增量日志模式，非全量覆写）
├── writing_rules.md     ← 写作偏好与避坑指南（自动进化）
└── draft_v*.md          ← 历史起草版本
```

## Workflow 1：记忆积累与融合

**触发词**：「扫描」「更新记忆」「处理新文档」「把XX加入知识库」

### 执行步骤

**Step 1 — 扫描目标目录（增量模式）**

- 列出用户指定目录下所有 `.pdf` / `.docx` / `.doc` / `.md` / `.txt` 文件
- 读取知识库 `index.json`，对每个文件计算 SHA256 哈希
- 三种判定结果：
  - **新文件**：`file_name` 不在 index.json 中 → 进入 Step 2 全量处理
  - **已修改**：`file_name` 存在但哈希不同 → 进入 Step 2 重新生成卡片，旧卡片保留为 `xxx.archived.md`
  - **未变化**：`file_name` 存在且哈希相同 → 跳过
- 可调用 `scripts/extract_docs.py` 辅助文本提取和哈希计算

**Step 2 — 逐文档生成知识卡片**

对每个新文件或已修改文件，读取其文本内容，在 `cards/` 目录生成独立的 MD 卡片。

卡片必须包含以下结构：

```markdown
# [原文档标题]

## 基本信息
- **文档名称**：xxx
- **来源文件**：xxx.docx
- **文档类型**：[工作部署/战略规划/培训材料/政策文件/研究报告/...]
- **日期**：YYYY-MM-DD
- **字数**：N字

## 核心主旨
[50字以内一句话概括]

## 关键内容提炼
[结构化摘要，保留数据、政策依据、核心论点]

## 可引用素材
[数据/指标、可直接引用的句子、场合/受众信息]

## 隐性关联提示
[本文与其他文档的关联、可支撑的写作场景]
```

**Step 3 — 更新 index.json**

以合法 JSON 格式追加或更新新卡片的元数据。

**Step 4 — 增量更新 macro_vision.md**

采用增量日志模式，不全量覆写。仅追加新发现，不删除旧条目。

**Step 5 — 反馈用户**

```
✅ 本次处理了 X 份文档（新增 N / 更新 M / 跳过 K）
📡 增量洞察：[简述新发现的跨文档关联]
```

## Workflow 2：智能起草

**触发词**：「起草」「帮我写」「写一份关于…」「生成一份…报告/方案/简报」

### 执行步骤

**Step 1 — 需求解析**

若信息不足，询问用户（最多2个问题）：主题与受众、核心论点或特别强调的方向。

**Step 2 — 两级漏斗筛选**

- 读取 `index.json`
- 基于关键词和标签，选出最相关的 Top 5～10 份文档
- 先输出"📌 素材定位分析"，列出选中文档及选择理由

**Step 3 — 精准加载卡片**

- 读取选中卡片，重点提取「可引用素材」
- 同时读取 `macro_vision.md` 和 `writing_rules.md`

**Step 4 — 生成结构化初稿**

- 严格结合 `writing_rules.md` 中的规则起草
- 素材以原文形式嵌入初稿，不保留内部引用标注
- 无材料支撑时标注 `[需补充材料]`

**Step 5 — 保存草稿**

- 新草稿保存为 `draft_v{n+1}.md`
- 向用户展示完整草稿内容

## Workflow 3：反馈修订与规则进化

**触发词**：「修改」「反馈」「调整一下」「这里改成…」

### 执行步骤

**Step 1 — 读取上下文**：最新版草稿 + macro_vision.md + writing_rules.md

**Step 2 — 修订草稿**：严格遵循用户修改指令，保存递增版本号

**Step 3 — 规则压缩与进化**：分析修改意见是否代表通用偏好，若是则更新 writing_rules.md（保持 ≤10 条核心规则）

**Step 4 — 反馈**：告知草稿已修订，新规则已存入规则库

---

## Workflow 4：知识库整理（全量重构）

**触发词**：「整理知识库」「重构全局观」「知识库有点乱」

全量重读所有卡片，重新归纳主题簇，重写 macro_vision.md（旧版归档），保留历史增量日志。

---

## 严格约束

1. **绝不使用向量数据库**，仅依赖纯文本读写和 LLM 上下文理解
2. **绝不编造引用**，所有注入初稿的素材必须真实存在于 index.json 和 cards/ 中
3. **保持 JSON 格式严谨**，更新 index.json 前必须先读取现有内容
4. **一文档一 MD**，禁止将所有卡片合并到单一大文件
5. **增量优先**，macro_vision.md 日常操作仅增量追加（Workflow 4 除外）
6. **卡片必须保留可引用素材**，数据和句子不可模糊化
