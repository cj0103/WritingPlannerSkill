# Writing Planner Skill — 政务写作提示词生成器

## 核心定位

**WritingPlannerSkill 是提示词工程层，smart-doc-memory 是知识材料层。两者是上下游关系：**

```
smart-doc-memory（材料层）        WritingPlannerSkill（提示词层）
─────────────────────────        ─────────────────────────
知识卡片 (cards/)         ──→    精准素材注入
全局索引 (index.json)     ──→    两级漏斗筛选
全局观 (macro_vision.md)  ──→    战略框架与叙事线
写作规则 (writing_rules.md) ──→  规则约束嵌入
                                    ↓
                          输出：结构化写作提示词
                          用途：喂给任意大模型生成公文
```

本技能**不直接生成最终公文**，而是产出一个最优的写作提示词（prompt）。这个提示词：
- 已注入精准素材（从知识卡片提取的数据、观点、叙事线）
- 已嵌入写作规则（文种结构、标准表述、受众适配、禁忌）
- 已明确输出要求（篇幅、格式、风格）
- 可直接复制粘贴到任何大模型中使用

## 当前知识库

本实例基于用户自建专题知识库（`knowledge_base/`）：
- 结构化知识卡片（`cards/`）
- 全局索引（`index.json`）
- 全局观图谱（`macro_vision.md`）
- 写作规则库（`writing_rules.md`）

四类核心写作场景：部署讲话、总结报告、实施方案、即兴简报。

## Workflow：生成写作提示词

### 完整流程

```
  ┌───────────────── Agent A（撰写员）────────────────┐    ┌─ Agent B（审核员）─┐
  │ 需求解析 → 两级漏斗筛选 → 素材拆解适配 → 加载    │    │                    │
  │ 规则 → 拼装提示词 → 保存交付                       │    │  10项检查清单逐项   │
  └──────────┬────────────────────────────────────────┘    │  审核               │
             │ 退修单（修订说明）                           │                    │
             │ ←───────────────────────────────────────────┴── 全部 ✅ → 交付 ──
             │                                                  ⚠️/❌ → 出具退修单
             │
             └── 收到退修单 → 按修订要求重新执行对应步骤 ──→ 第N+1轮
                 (最多 3 轮迭代，超限后标注未达标项交付)
```

**触发词**：「生成提示词」「写一个prompt」「写作规划」「帮我规划一篇」「给我一个XX的提示词」「怎么让AI写一篇XX」

### 执行步骤

**Step 1 — 需求解析**

从用户输入中提取以下要素（若缺失则提问，最多2个）：

| 要素 | 说明 | 示例 |
|------|------|------|
| 文种 | 部署讲话 / 总结报告 / 实施方案 / 即兴简报 | "部署讲话" |
| 主题 | 具体写作主题 | "2026年下半年金融科技工作部署" |
| 受众 | 领导 / 条线干部 / 企业 / 公众 | "局内干部" |
| 场合 | 具体使用场景 | "半年度工作会议" |
| 特殊要求 | 字数、侧重点、敏感话题等 | "重点讲可信数据空间进展" |

**Step 2 — 两级漏斗筛选卡片**

- 读取 `knowledge_base/index.json`
- 基于文种和主题，从主题簇中确定目标簇
- 参考 `macro_vision.md` 中的「写作场景 × 文档映射」，锁定必引和选引卡片
- 读取对应卡片全文（不仅仅是摘要）
- 输出"📌 素材定位"，列出选中卡片及选择理由

**Step 2.5 — 素材拆解与适配（关键步骤）**

这是本技能区别于简单"搬运卡片内容"的核心环节。素材不是从卡片里整体搬过来的，而是逐条拆解、逐条适配的。

**2.5.1 逐条拆解**

从每张选中卡片的「可引用素材」和「关键内容提炼」板块中，将素材拆解为独立的知识单元。每条单元包含：
- 原始数据/表述（保留精确数字和完整口径）
- 来源卡片（可溯源）
- 原文种信息（该素材在原文档中的使用场景）

**2.5.2 文种适配标注**

对每条素材标注在当前文种下的用法。参考 gov-doc-annotator 的文种适配模式：

| 素材类型 | 部署讲话中的用法 | 总结报告中的用法 | 实施方案中的用法 | 即兴简报中的用法 |
|----------|-----------------|-----------------|-----------------|-----------------|
| 量化数据 | 作为成就展示，配合排比句式 | 作为成果佐证，保留原始口径 | 作为目标基线，说明增长空间 | 精选1-2个最有冲击力的数字 |
| 政策条款 | 作为工作依据，强化权威性 | 作为背景交代，简述即可 | 作为上位法依据，必须标注文号 | 一般不引用，除非是关键政策 |
| 叙事线 | 全文贯穿，定调主线 | 回顾部分引用，展望部分呼应 | 背景依据部分引用 | 简化为1句核心判断 |
| 概念术语 | 需解释但不必过细 | 首次出现时给出定义 | 需精确定义，作为工作抓手 | 必须通俗化"翻译" |

**2.5.3 层级判断**

参考 gov-doc-annotator 的层级标注逻辑，对每条素材做三级分类：

- **正文引用**：区级数据/本部门直接相关 → 可作为论点主体
- **背景铺垫**：市级/宏观数据 → 仅用于开篇定调或对比参照，不超过全文10%
- **不适用**：与当前受众/场合不匹配 → 直接过滤，不进入提示词

**2.5.4 受众适配过滤**

根据 Step 1 确定的受众类型，对素材做适配标记：

- **非技术受众**（街道干部、企业代表、公众）：标记需要通俗化"翻译"的概念，在提示词中附带通俗解释建议
- **技术受众**（条线干部、数据局内部）：保留术语原貌，可适当加深技术细节
- **领导汇报**：精简数据，突出判断和方向，数据作为佐证而非主体
- **对外宣讲**：仅使用可公开数据，标记内部数据为"不可用"

**2.5.5 场合适配过滤**

- **内部会议**：可使用未公开数据、内部讨论观点
- **对外交流**：仅使用已公开数据、正式发布的口径
- **上级汇报**：数据需经过核实，标注"以最终发布口径为准"
- **街道交流**：减少层级概念，增加协作场景的案例

**2.5.6 输出素材适配表**

完成拆解和适配后，输出一张素材适配表，作为 Step 4 拼装提示词的核心输入：

```markdown
## 📊 素材适配表

| # | 原始素材 | 来源卡片 | 层级 | 当前文种用法 | 受众处理 | 是否注入 |
|---|---------|---------|------|------------|---------|---------|
| 1 | GDP 3615.72亿元 | card_06 | 正文引用 | 开篇成就展示 | 直接使用 | ✅ |
| 2 | "一核两翼三支撑" | card_04 | 正文引用 | 全文框架主线 | 直接使用 | ✅ |
| 3 | 市级金融科技指标 | card_02 | 背景铺垫 | 开篇1句带过 | 直接使用 | ✅ |
| 4 | 可信数据空间技术架构 | card_07 | 正文引用 | — | 需通俗化：比喻为"数据的银联" | ✅ |
| 5 | 内部招商数据 | card_07 | 不适用 | — | 对外交流不可用 | ❌ |
```

**Step 3 — 加载写作规则**

- 读取 `writing_rules.md`
- 提取与当前文种相关的：结构规则、标准表述、受众适配规则、禁忌

**Step 4 — 拼装提示词**

按照以下模板生成结构化提示词。**关键原则**：提示词中的「核心素材」板块不是从卡片搬运原文段落，而是将 Step 2.5 的素材适配表展开为可直接指导写作的结构化内容——每条素材都带有当前文种下的具体用法说明。

```markdown
# 写作提示词：[主题]

## 角色设定
你是一位 [角色描述]，需要撰写一份 [文种名称]。

## 写作背景
[从 macro_vision.md 提取的战略定位、当前形势]

## 结构要求
[从 writing_rules.md 提取的该文种标准结构，含各部分要点]

## 核心素材（已按当前文种和受众适配）

### 正文引用素材（可直接作为论点主体）
[逐条列出，每条包含：数据/表述原文 + 在当前文种中的具体用法说明 + 建议放置位置]

示例：
- **数据**："地区生产总值达到3615.72亿元"
  **用法**：作为"[本区]有基础"段落的支撑数据，放在开篇定调之后
  **位置**：第一部分"工作回顾"第2段

### 背景铺垫素材（仅用于定调或对比，不超过全文10%）
[逐条列出，标注：原文 + 用法 + 篇幅限制]

### 需要通俗化处理的概念（非技术受众场景）
[逐条列出，标注：原术语 + 建议通俗解释/比喻]

示例：
- **原术语**："可信数据空间"
  **建议解释**："可以理解为数据的银联——不同机构的数据可以在一个安全可信的环境里流通和使用"

### 关键叙事线
[从 macro_vision.md 提取的可复用叙事线，标注在全文中的贯穿方式]

### 标准表述
[从 writing_rules.md 提取的核心概念标准表述，标注不可替换的关键词]

## 写作规则
### 风格要求
[语言风格、篇幅要求]

### 受众适配
[面向当前受众的特殊要求，包括需要通俗化的概念列表]

### 禁忌事项
[写作禁忌，与当前文种/受众相关者]

## 输出要求
- 篇幅：[N]字左右
- 格式：[分段要求]

> 注意：提示词中的素材已直接以原文形式嵌入，下游大模型无需也无法访问卡片源文件。每条素材都附带了在当前文种/受众下的具体用法说明，大模型应严格遵循这些用法说明而非自由发挥。
```

**Step 5 — 保存并交付**

- 提示词保存到 `knowledge_base/prompts/` 目录，文件名为 `prompt_[主题]_YYYYMMDD.md`
- 向用户展示完整提示词
- 附带使用说明："将此提示词复制到任意大模型对话中，即可生成初稿"

**Step 6 — 双 Agent 审核循环**

本步骤将"自我纠偏"升级为**双角色协作**——模拟真实公文写作中的"撰稿—核稿"机制。Step 1~5 由 **Agent A（撰写员）** 执行，Step 6 由 **Agent B（审核员）** 接管。B 审核不通过时，出具结构化"退修单"，退回 A 修改后再审。

**6.0 角色定义**

| 角色 | 职责 | 负责的步骤 | 核心原则 |
|------|------|-----------|---------|
| **Agent A（撰写员）** | 撰写提示词，接收退修单并修改 | Step 1 → Step 2 → Step 2.5 → Step 3 → Step 4 → Step 5 | 以 B 的退修单为唯一修改依据，不自作主张 |
| **Agent B（审核员）** | 审核提示词质量，出具退修单 | Step 6.1 → Step 6.2 → Step 6.4 | 只审不写，保持独立判断 |

> **关键规则**：Agent A 和 Agent B 是同一技能执行过程中切换的两个角色，**但切换必须显式声明**。当执行到审核环节时，必须说"现在切换为 **Agent B（审核员）** 角色"，审核完毕回到撰写环节时，必须说"现在切换回 **Agent A（撰写员）** 角色，开始第 N 轮修改"。角色的显式切换确保两个"头脑"互不干扰——A 专注于"怎么写"，B 专注于"审什么"。

**6.1 检查清单（Agent B 的审核工具）**

B 对 A 交付的提示词逐项核查：

| # | 检查项 | P0/P1 | 检查方法 | 通过标准 |
|---|-------|-------|---------|---------|
| C1 | 素材是否逐条拆解而非整段搬运 | P0 | 检查「核心素材」板块，确认每条素材是独立数据单元而非卡片段落截取 | 每条素材 ≤ 3 句，且用法说明独立存在 |
| C2 | 每条素材是否标注了当前文种的用法 | P0 | 检查每条正文引用素材是否包含「用法」说明 | 100% 的正文引用素材都有用法标注 |
| C3 | 层级判断是否执行 | P0 | 检查素材是否分为「正文引用」和「背景铺垫」两大类 | 两个分类在提示词中都有表现 |
| C4 | 文种模板是否正确匹配 | P0 | 检查提示词结构与目标文种的标准模板是否一致 | 结构要素完全匹配，非该文种要素不存在 |
| C5 | 受众适配是否处理 | P1 | 检查非技术受众场景是否含通俗化概念板块 | 若有非技术受众，必须存在「需要通俗化处理的概念」板块 |
| C6 | 场合适配是否处理 | P1 | 检查对外交流场合是否有内部数据标记 | 对外场合必须标注内部数据为"不可用" |
| C7 | 规则是否准确嵌入 | P1 | 检查 writing_rules 中的关键标准表述是否出现在提示词中 | 无遗漏，无泛化 |
| C8 | 提示词是否独立可用 | P1 | 检查提示词是否包含素材原文（不依赖外部文件） | 所有数据/表述在提示词中都有原文，无「参见 card_xx」类引用 |
| C9 | 素材适配表是否完整生成 | P2 | 检查 Step 2.5 是否产出了素材适配表 | 表含原始素材、来源卡片、层级、文种用法、受众处理、是否注入六列 |
| C10 | 背景铺垫素材是否不超过全文10% | P2 | 粗略估算背景素材在全文字数中的占比 | 背景素材篇幅占比 ≤ 10% |

**6.2 审核结论（Agent B 的判定）**

B 对每一项给出判定，并汇总为整体结论：

| 结论 | 条件 | 后续操作 |
|------|------|---------|
| ✅ **通过** | 全部 10 项为 ✅ | 交付提示词 |
| ⚠️ **微调后通过** | 仅 P2 项不通过，或 P1 项 ≤ 2 个且为轻微瑕疵 | 出具退修单，退回 A 做定向微调 |
| ❌ **退回重写** | 任一 P0 项不通过，或 P1 项 ≥ 3 个，或 P2 项大面积不通过 | 出具退修单，退回 A 重做 Step 2.5 或 Step 4 |

**6.3 双 Agent 协作流程**

```
     ┌─────────────────────────────────────────────────────────┐
     │ 第 1 轮                                                 │
     │ Agent A（撰写员）: 执行 Step 1~5，生成初版提示词        │
     └────────────────────────┬────────────────────────────────┘
                              │
     ┌────────────────────────▼────────────────────────────────┐
     │ Agent B（审核员）: 执行 6.1~6.2 审核                    │
     │ 结论: ✅ 通过 ──→ 交付，流程结束                        │
     │       ⚠️/❌ ──→ 执行 6.4，出具退修单，进入第 2 轮      │
     └────────────────────────┬────────────────────────────────┘
                              │ 退修单
     ┌────────────────────────▼────────────────────────────────┐
     │ 第 2 轮                                                 │
     │ Agent A（撰写员）: 收到退修单 → 定向修改 → 生成修订版  │
     └────────────────────────┬────────────────────────────────┘
                              │
     ┌────────────────────────▼────────────────────────────────┐
     │ Agent B（审核员）: 再次审核                             │
     │ 结论: ✅ 通过 ──→ 交付                                  │
     │       ⚠️/❌ ──→ 出具退修单，进入第 3 轮                │
     └────────────────────────┬────────────────────────────────┘
                              │ 退修单
     ┌────────────────────────▼────────────────────────────────┐
     │ 第 3 轮                                                 │
     │ Agent A（撰写员）: 收到退修单 → 定向修改 → 生成修订版  │
     └────────────────────────┬────────────────────────────────┘
                              │
     ┌────────────────────────▼────────────────────────────────┐
     │ Agent B（审核员）: 再次审核                             │
     │ 结论: ✅ 通过 ──→ 交付                                  │
     │       ⚠️/❌ ──→ 输出当前版本 + 标注未达标项 + 交付     │
     └─────────────────────────────────────────────────────────┘
```

**6.4 Agent B 出具退修单**

B 审核不通过时，出具标准格式的退修单，退回给 A：

```markdown
## 🔄 退修单 — 第 N 轮

### 退修结论
[⚠️ 微调后通过 / ❌ 退回重写]

### 不通过项明细
| 检查项 | 问题描述 | 原因分析 |
|--------|---------|---------|
| C2 | 素材"GDP 3615.72亿"缺少用法说明 | A 在 Step 2.5 拆解时遗漏了用法标注 |
| C5 | 未添加「需要通俗化处理的概念」板块 | 受众为街道干部，属于非技术受众，A 未按 Step 2.5.4 执行受众适配 |

### 回退至步骤
- C1/C2/C3 → 回退到 **Step 2.5**（素材拆解与适配）
- C4/C5/C6/C7/C8/C10 → 回退到 **Step 4**（拼装提示词）
- 本案例：Step 2.5（C2 需要重做素材拆解）+ Step 4（C5 需要新增板块）

### 具体修订要求
1. 为素材"GDP 3615.72亿"补充用法说明——在 Step 2.5 中重新拆解该素材，标注"作为[本区]成就展示数据，放在开篇"
2. 在「核心素材」中新增「需要通俗化处理的概念」板块——至少包含：可信数据空间、区块链、数据要素

### 本轮重点
本次审核重点关注 C2 和 C5，其他已通过的项不可回退。
```

**退修单撰写规则（Agent B 必须遵守）：**
1. **问题必须定位到具体素材/板块**，不能笼统说"素材有问题"
2. **必须指明回退的具体步骤**（Step 2.5 还是 Step 4）
3. **修订要求必须可操作**——A 拿到就能直接改，不需要猜测
4. **标注本轮审核重点**——已通过的项不应回退，避免"开倒车"

**6.5 Agent A 收到退修单后的修改规范**

A 收到退修单后，按以下规范执行修改：

1. **严格以退修单为准**：只修改退修单中指出的问题，不额外发挥
2. **回退到指定步骤**：退修单说回 Step 2.5，就回到 Step 2.5 重新执行（含其后所有步骤）；说回 Step 4，就只改 Step 4
3. **保留已通过的项**：前一轮已通过的 C1/C3/C4 等不做任何改动
4. **完成修改后标记**：在提示词头部附加 `[第 N 轮修订版]` 标记，然后提交给 B 再次审核
5. **若对退修单有疑问**：仍按退修单执行修改，可在交付时附加注释说明

**6.6 失败预防与知识沉淀（长期积累）**

每次双 Agent 协作循环完成后，沉淀一条经验到以下表格中。随使用增加，逐步降低退修率：

| 常见退修原因 | 根因定位 | 预防措施（更新到对应步骤） |
|-------------|---------|--------------------------|
| 素材未拆解（C1 退修） | A 在 Step 2.5 执行仓促 | Step 2.5 增加自我校验："拆解后检查：每条素材是否已独立化" |
| 受众适配遗漏（C5 退修） | Step 1 中受众未明确提取 | Step 1 增加受众确认环节并写入工作记录 |
| 标准表述遗漏（C7 退修） | A 读取 writing_rules.md 不完整 | Step 3 增加逐条核对机制，而非通读 |

*Agent B 发现新的退修模式时，主动补充到此表。*

## 严格约束

1. **只生成提示词，不生成最终公文**：本技能的产出是 prompt，不是成品文稿
2. **素材必须逐条拆解适配，不可整段搬运**：卡片中的素材必须先拆解为独立数据单元，经过文种适配、层级判断、受众过滤后，才能注入提示词。直接复制卡片段落到提示词中是违规的
3. **每条素材必须标注在当前文种中的用法**：不只是"有哪些素材"，更重要的是"在当前文种和受众下怎么用"
4. **规则必须准确嵌入**：写作规则不能泛化或遗漏，每条规则要体现到 prompt 的具体段落中
5. **提示词必须独立可用**：产出的 prompt 应包含足够完整的上下文（素材原文 + 用法说明），可直接复制到其他大模型使用，不依赖卡片源文件
6. **文种模板不可混用**：部署讲话的结构不能套用到实施方案
7. **层级判断必须执行**：正文引用 vs 背景铺垫 vs 不适用，三层分类不可跳过。背景铺垫素材不超过全文10%
8. **必须执行双 Agent 审核循环**：生成的提示词必须经过 Agent B 的审核，不合格项通过退修单退回 Agent A 修改。最大循环次数为 3 次，第 3 次后仍有未达标项须向用户标注。角色切换必须显式声明

## 与 smart-doc-memory 的分工

| 维度 | smart-doc-memory | WritingPlannerSkill |
|------|-----------------|---------------------|
| 定位 | 知识材料层 | 提示词工程层 |
| 输入 | 原始文档（PDF/DOCX） | 用户写作需求 + 知识库 |
| 核心产出 | 知识卡片、索引、全局观、规则库 | 结构化写作提示词 |
| 素材处理 | 从文档中提取并结构化（保留原貌） | 逐条拆解 + 文种适配 + 层级判断 + 受众过滤 |
| 最终产物 | 材料（可读、可引用） | 指令（可喂给AI） |
| 触发词 | 扫描、更新记忆、起草 | 生成提示词、写prompt、写作规划 |
| 对下游 | 提供精准素材 | 产出的prompt → 任意大模型 → 最终公文 |

## 设计理念：前馈 + 反馈的闭环体系

writing-planner 不是简单的"卡片内容搬运工"。如果直接把卡片段落复制到提示词里，下游大模型拿到的是未经加工的原材料，它不知道这段数据在当前文种下应该怎么用、放在哪里、是否需要通俗化处理。

**前馈通道**（Step 1 → Step 5）：素材拆解适配（Step 2.5）做的是"把原材料加工成半成品"的工作——不是替大模型写作，而是替大模型做好写作前的准备工作。这个步骤的设计借鉴了 gov-doc-annotator 的标注级理解深度：每条数据不再是文档的一个片段，而是一个独立的知识单元，带有文种适配说明、层级判断和受众处理建议。

**反馈通道**（Step 6）：双 Agent 审核循环将"自检"升级为"他检"——Agent B 以独立的审核视角审查 A 的产出，出具退修单，退回 A 修改。这模拟了真实公文写作中的"撰稿—核稿"机制，比自我检查更严格：A 专注于"怎么写"，B 专注于"审什么"，两个角色互不干扰。退修单制度确保了每次修改都有明确的问题指向和操作指引，避免"凭感觉改"。整体采用 PDCA（Plan-Do-Check-Act）闭环思想：Step 1~5 是 Do，Step 6.1~6.2 是 Check，Step 6.4~6.5 是 Act。

**进化机制**：Step 6.6（失败预防与知识沉淀）记录了每次双 Agent 协作循环中出现的退修原因，随着使用积累，逐步降低退修率，让技能越用越准。
