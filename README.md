# WritingPlannerSkill

> 让 AI 帮你写提示词，而不是帮你写文章。

## 一句话说清

**WritingPlannerSkill 是一套政务写作 AI 技能组合。它不直接生成公文，而是通过两阶段流水线——先让 AI 读懂你的工作文档、建立知识库，再根据写作需求生成高质量的结构化提示词——让你把提示词喂给任意大模型，得到精准的初稿。**

它解决的核心矛盾：大模型不会写政府公文，不是因为文笔不好，而是因为它不知道你的工作背景、政策语境和组织语言体系。WritingPlannerSkill 把"背景知识注入"这件事做成了一套可复用的流程。

## 当前状态

两个核心技能已完成多次迭代并投入实际使用，知识库管道已验证通过：

- **smart-doc-memory**：支持 10+ 份文档的全量/增量知识库化，自动生成结构化卡片、索引、全局观图谱和写作规则库。已实现 SHA256 哈希增量判定，支持 PDF/DOCX/DOC/XLSX 多格式提取。
- **writing-planner**：已实现完整的 **6 步闭环写作工作流**（Step 1 需求解析 → Step 2 两级漏斗筛选 → Step 2.5 素材拆解适配 → Step 3 加载规则 → Step 4 拼装提示词 → Step 5 保存交付）。核心亮点：
  - **Step 2.5「素材拆解适配」**：不搬运卡片段落，而是逐条拆解素材，按文种、层级、受众、场合四维标注后再注入提示词
  - **Step 6「双 Agent 审核循环」**：模拟真实公文"撰稿—核稿"机制。Agent A（撰写员）执笔，Agent B（审核员）独立审核，不合格项通过结构化退修单退回重写，最多 3 轮迭代
  - 已在街道交流、工作部署、产业推介交流会等真实场景验证，通过实际公文打磨过程持续优化

## 架构：两层技能，一条闭环流水线

```
smart-doc-memory（材料层）           writing-planner（提示词层）
─────────────────────────           ─────────────────────────
扫描原始文档                        读取用户写作需求
    ↓                                    ↓
生成知识卡片 + 索引                两级漏斗筛选素材
    ↓                                    ↓
构建全局观图谱                    素材逐条拆解与四维适配
    ↓                                    ↓
写作规则库自动进化                加载写作规则
    ↓                                    ↓
【产出：结构化知识】              拼装结构化提示词
                                       ↓
                              ┌─ Agent B（审核员）审核 ─┐
                              │  10项检查清单逐项核查    │
                              └──────────┬──────────────┘
                                全部通过 │  不通过
                                    ↓    │    ↓
                              【交付】   │  出具退修单
                                        │    ↓
                                        └→ Agent A 按单修改
                                          (最多 3 轮迭代)
```

### smart-doc-memory：让 AI 读懂你的工作文档

- 扫描目录，把 PDF/DOCX/DOC 等文档转为结构化知识卡片（一文档一 MD）
- 自动构建全局索引（`index.json`）和跨文档关联图谱（`macro_vision.md`）
- 根据你的修改反馈自动进化写作规则库（`writing_rules.md`）
- 支持增量更新（SHA256 哈希比对，只处理新文档和已修改文档）

### writing-planner：让 AI 帮你规划写作

- 解析你的写作需求（文种、主题、受众、场合）
- 从知识库精准筛选素材（两级漏斗：索引 → 卡片）
- **素材拆解适配（核心差异化能力）**：逐条拆解卡片素材为独立知识单元，按文种/层级/受众/场合四维标注，过滤不适用的素材，标注需要通俗化"翻译"的概念
- 按文种模板拼装结构化提示词（嵌入原文素材 + 用法说明 + 叙事线 + 标准表述 + 规则约束）
- 提示词独立可用——复制到任何大模型即可生成初稿
- 支持四种文种：部署讲话、总结报告、实施方案、即兴简报
- **双 Agent 审核循环**：Agent B 按 10 项检查清单逐项审核，出具结构化退修单，Agent A 按单修改。模拟"撰稿—核稿"机制，确保输出质量

### 素材拆解适配：不只是搬运

writing-planner 区别于简单 RAG 的关键设计。它不把知识卡片的内容整段复制到提示词里——而是：

1. **逐条拆解**：将卡片段落拆解为独立的数据单元（每条保留精确数字和完整口径）
2. **文种适配**：同一条数据在部署讲话中是"成就展示"，在实施方案中是"目标基线"，在即兴简报中则精选 1-2 个冲击力数字
3. **层级判断**：区级数据→正文引用（作为论点主体）；市级/宏观数据→背景铺垫（不超过全文 10%）；不适配的数据→直接过滤
4. **受众过滤**：非技术受众自动标记通俗化概念并附带比喻建议；对外交流过滤内部数据
5. **场合适配**：内部会议/对外交流/上级汇报/街道交流四种场合各有适配规则

最终产出的是"半成品"——不是替大模型写作，而是替大模型做好写作前的所有准备工作。

## 与 RAG 的区别

| | RAG | WritingPlannerSkill |
|---|---|---|
| **检索方式** | 向量相似度检索 | 结构化索引 + 关键词 + 主题簇 |
| **知识粒度** | 文本块（chunk） | 知识卡片（完整文档摘要 + 可引用素材 + 跨文档关联） |
| **上下文控制** | 黑盒——你无法预知检索回了什么 | 白盒——两级漏斗筛选，每一步你都看得见 |
| **写作能力** | 检索 + 生成一步到位 | 先规划再执行：提示词工程 + 任意模型 |
| **模型依赖** | 绑定特定 RAG 管道 | 不绑定任何模型，prompt 可喂给 Claude/GPT/DeepSeek 等 |
| **知识进化** | 需要重新 embedding | 增量更新卡片 + 写作规则自动进化 |
| **运维成本** | 向量数据库 + embedding 服务 | 纯文本文件，零外部依赖 |
| **适用场景** | 开放域问答、客服、知识检索 | 封闭域写作（公文、报告、方案），对精确性要求高 |

**核心哲学差异**：RAG 的思路是"把知识喂给模型，让模型自己判断怎么用"；WritingPlannerSkill 的思路是"你先帮模型把知识组织好、规则理清楚，再给它一个精准的写作指令"。前者追求覆盖面，后者追求精确性。

## 与 LLM Wiki 的区别

LLM Wiki（以 Karpathy 的设计为代表）的核心思想是"用 LLM 来写 Wiki 条目"，本质是**知识管理工具**——让 AI 帮你把散落的信息整理成结构化的知识页面。

WritingPlannerSkill 继承了这个思路（知识卡片就是 Wiki 条目），但往前多走了一步：**知识不仅要被管理，还要被调度到写作场景中**。

| | LLM Wiki | WritingPlannerSkill |
|---|---|---|
| **核心产物** | 知识页面 | 知识页面 + 写作提示词 |
| **知识组织** | 按主题归档 | 按主题簇 + 写作场景双重索引 |
| **下游用途** | 人阅读、查阅 | 人阅读 + 喂给 AI 写作 |
| **叙事线** | 无 | 有——从文档中提炼可复用的叙事线（战略定位型、任务部署型等） |
| **规则系统** | 无 | 有——写作规则库随反馈自动进化 |
| **跨文档关联** | 超链接 | 隐性关联提示 + 全局观图谱 |
| **对写作的支持** | 间接——你读了 Wiki 自己写 | 直接——生成 prompt，AI 帮你写初稿 |

可以这样理解：LLM Wiki 是图书馆，WritingPlannerSkill 是图书馆 + 写作辅导老师——前者帮你找到书，后者帮你把书里的东西变成文章。

## 快速开始

### 前置条件

- Python 3.8+
- 安装依赖：`pip install pdfplumber PyPDF2 python-docx olefile`
- 用于 `.doc` 文件提取（可选）：安装 antiword

### 第一步：构建知识库

1. 把你的工作文档放到一个目录中（支持 PDF、DOCX、DOC、MD、TXT）
2. 在 WorkBuddy 中触发：`扫描 [你的文档目录]`
3. smart-doc-memory 会自动提取文本、生成知识卡片、构建索引和全局观

### 第二步：生成写作提示词

在 WorkBuddy 中描述你的写作需求，例如：

> 我需要去街道交流，一是介绍某项产业工作，二是介绍本单位的业务职能。帮我生成一个讲稿提示词。

writing-planner 会：
1. 解析需求（文种、主题、受众、场合）
2. 从知识库筛选相关素材
3. 加载写作规则
4. 输出一份结构化的写作提示词

### 第三步：喂给大模型

把生成的提示词复制到 Claude、GPT-4、DeepSeek 等任意大模型，得到初稿。然后根据实际情况微调。

## 项目结构

```
WritingPlannerSkill/
├── README.md                          ← 你正在看
├── .gitignore                         ← 排除知识库内容和敏感数据
├── scripts/
│   └── extract_docs.py                ← 文档文本提取脚本（含哈希计算）
└── .workbuddy/skills/
    ├── smart-doc-memory/SKILL.md      ← 材料层技能定义
    └── writing-planner/SKILL.md       ← 提示词层技能定义
```

> 知识库内容（`knowledge_base/`）不在仓库中——你需要用自己的文档构建。两个 SKILL.md 提供完整的工作流和模板，开箱即用。

## 适用场景

- **政府/事业单位**：部署讲话、总结报告、实施方案、即兴简报等公文写作
- **政策研究机构**：专题研究报告、政策建议、调研报告
- **企业内部**：战略规划、项目方案、述职报告等需要大量背景知识的写作场景

核心特征：**写作素材分散在多份文档中，需要对素材进行精准调度和规则约束的场景**。

## 未来改进方向

### 短期（降低使用门槛）

- **一键知识库初始化向导**：引导用户配置文档目录、写作场景、偏好规则，交互式完成首次构建
- **提示词效果对比**：同一个写作需求，生成 2-3 个不同侧重点的提示词，让用户选择
- **写作规则可视化编辑**：不用手动改 MD，提供规则项的勾选/编辑界面
- **用户写作偏好自动学习**：从用户对稿件的修改反馈中自动学习风格偏好（如偏好的口语化程度、段落结构习惯），避免每次都需要人工调整

### 中期（增强智能）

- **自动发现写作规则**：从用户的修改反馈中自动归纳规则，不需要人工分析。当前 Workflow 3 的规则进化需要 AI 手动分析修改意见——可以更进一步，让系统自动识别"用户反复修改的地方"并提炼为规则
- **叙事线自动生成**：当前叙事线是人工从文档中提炼的。可以探索让 AI 自动从知识卡片中识别可复用的叙事模式
- **多知识库切换**：支持同时管理多个专题知识库，写作时自动判断应该用哪个库
- **受众画像系统**：为不同受众建立写作偏好画像（领导偏好数据型还是案例型？条线干部需要多细的颗粒度？），写作时自动适配
- **审核策略自适应**：根据历史退修记录，Agent B 自动调整审核重点——高频退修项提高权重，从未出问题的项降低检查频率

### 长期（范式突破）

- **写作数字孪生**：积累足够的写作反馈后，训练一个"你的写作风格模型"——不是通用大模型，而是专门模仿你个人写作风格的小模型。提示词不再是"写一份部署讲话"，而是"以我的风格写一份部署讲话"
- **反向知识提取**：不仅从文档提取知识，还从你写过的公文反向提取隐性知识——你的论证习惯、常用的起承转合模式、偏好的数据呈现方式——这些隐性知识当前没有被结构化
- **跨组织写作协作**：多个用户的写作规则库可以互相参考和融合，形成"组织写作规范"——新同事入职时，不是看一叠写作规范文档，而是直接使用已经被规则库约束的提示词系统
- **写作影响评估**：写完一份公文后，系统自动分析这份公文可能对后续工作产生什么影响——引用了哪些政策？涉及哪些部门？与哪些既有工作存在交叉？把写作从"输出文档"升级为"影响预判"

### 技术架构改进

- **卡片格式标准化**：当前知识卡片是 Markdown，结构靠约定。可以定义 JSON Schema，让卡片元数据可被程序化查询
- **提示词 A/B 测试框架**：系统化地比较不同提示词策略的写作效果，积累"什么提示词策略对什么文种最有效"的经验
- **插件化模板系统**：当前四种文种模板是硬编码在 SKILL.md 中的。可以设计模板插件机制，让用户自定义和分享模板

## 隐私与安全

### 技能文件隐私审计

两个核心技能文件（`smart-doc-memory/SKILL.md` 和 `writing-planner/SKILL.md`）均为方法论和工作流描述，**不包含个人隐私信息**（姓名、职务、文件路径、系统用户名、IP 地址等）。模板中使用通用占位符（如"[某区数据局]"），不包含任何真实组织名称。

### 知识库数据隔离

- 知识库内容（`knowledge_base/`）**不在仓库中**，由 `.gitignore` 排除
- 工作记忆（`.workbuddy/memory/`）**不在仓库中**，同样被排除
- 每个用户需要用自己的文档构建知识库，不会交叉污染
- 所有数据处理在本地完成，不需要上传到任何外部服务

### 推送范围

仓库仅包含以下可公开的文件：
- `README.md` — 项目说明
- `.gitignore` — 排除规则
- `scripts/extract_docs.py` — 文本提取工具脚本
- `.workbuddy/skills/smart-doc-memory/SKILL.md` — 材料层技能定义
- `.workbuddy/skills/writing-planner/SKILL.md` — 提示词层技能定义

**不会推送的内容**：知识卡片、索引、全局观图谱、写作规则库、生成的提示词、工作记忆。

## 设计哲学

1. **简单胜过复杂**。没有向量数据库，没有 embedding，没有 API 调用。纯文本文件 + LLM 上下文理解。
2. **白盒胜过黑盒**。每一步——从素材筛选到规则嵌入——用户都看得见。你不会被一个不知道从哪里检索来的文本块左右写作方向。
3. **分离关注点**。材料管理和提示词工程是两个独立的问题，不应该耦合在一起。smart-doc-memory 管材料，writing-planner 管指令。
4. **闭环迭代**。采用 PDCA（Plan-Do-Check-Act）思想。前馈通道（Step 1~5）生成，反馈通道（Step 6）审核——不是生成完就交付，而是经过"撰稿—核稿—退修—再核"的闭环打磨。Agent A 和 Agent B 分工协作，模拟真实公文写作中的双人审核机制。
5. **渐进进化**。写作规则不是一次性定义好的，而是在每次修改反馈中自动进化的。退修记录沉淀为预防机制，用得越多，越贴合你的风格。
6. **模型无关**。最终写作可以用任何大模型完成。WritingPlannerSkill 只负责"把话说清楚"——告诉模型你要什么、有什么材料可以用、有什么规则要遵守。
